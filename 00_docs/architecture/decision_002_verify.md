# Decision 002: Level 2 Verify — majority voting по осям OCR

**Статус:** Реализовано.

## Контекст

ADR-001 зафиксировал интерфейс `verify()`, но отложил стратегию. Эмпирика multi-question OCR (см. сетку `chunk_size × max_workers` на тестовом документе) показала прямую мотивацию:

- `chunk_size=1` (одна сущность за запрос) — самый надёжный, самый медленный.
- `chunk_size≥3` (мультивопросный батч) — ~3–4× быстрее, но ~50% значений отличаются от baseline.
- Без независимого эталона нельзя сказать, какой прогон правильный.

Нужен механизм, который даёт по каждой сущности оценку уверенности и позволяет выбирать канонический ответ. При этом в рамках текущей инфраструктуры (один OCR-провайдер, один ключ API) допустимых «осей» для независимых прогонов немного: `chunk_size`, порядок промптов внутри чанка, DPI рендеринга, температура, модель. Самая дешёвая и наиболее релевантно ортогональная к ошибкам — `chunk_size`.

## Решение

`verify(pages, axes, max_workers)` выполняет N независимых OCR-прогонов с разными значениями `chunk_size` (ось varation), затем применяет majority voting по нормализованным значениям.

### Семантика

- На входе: `pages` (фильтр страниц, как у `scan`/`resolve`), `axes` — список `chunk_size` (дефолт `[1, 3, 5]` из env `OCR_VERIFY_AXES`), `max_workers`.
- Для каждой оси прогоняется полный OCR-проход по всем entries попадающих в `pages` (как `resolve`, но без записи в state). Результат каждого прогона — dict `{entity_id: {value, context, status}}`.
- Для каждой сущности собирается N голосов. Нормализация для сравнения: trim, collapse пробелов, lower; пустое/`None` → голос `__NO_DATA__`; ошибочный прогон (status=error) — не голосует.
- Выбирается winner: максимум голосов; при ничьей — более ранняя ось (первый прогон важнее).
- Запись в registry:
  - `value` = оригинальное (ненормализованное) значение первого прогона, выдавшего winner-голос; пустая строка, если winner — `__NO_DATA__`.
  - `context` — из того же прогона.
  - `confidence` = `"k/N"`, где `k` — голоса за winner, `N` — общее число проголосовавших (не полных осей; ошибки исключаются).
  - `verified` = `True` тогда и только тогда, когда `k == len(axes)` (единогласно, без ошибок).
  - `resolution` = `2`.

### Что НЕ делаем

- Не храним в state.json полные результаты всех прогонов (`samples`). Раздувает state, используется редко. При необходимости — CLI-флаг для дампа в отдельный файл (не в этой итерации).
- Не переиспользуем существующий `value` из предыдущего `resolve` как один из голосов. Голосуют только свежие прогоны. Причина: семантическая чистота — `verify` не зависит от того, какой `chunk_size` был у `resolve`.
- Не варьируем DPI/модель/температуру. Возможно позже, если эмпирика покажет, что `chunk_size` даёт слишком коррелированные ошибки.

### Гранулярность

Verify, как и Resolve, работает на уровне страницы. Страница получает `page_resolution="verified"`, когда для неё прошёл majority voting (хотя бы одна сущность верифицирована). Entity с unanimous verdict получает `verified=True`, иначе — `False` при `confidence<N/N`.

### Идемпотентность

Повторный `verify` с теми же осями не обязан давать одинаковый результат (OCR недетерминирован). Но каждый вызов полностью перезаписывает `value`/`context`/`confidence`/`verified` для целевых сущностей. Это сознательный trade-off: простота реализации важнее стабильности — если нужна стабильность, вызывающий может сохранять снэпшот registry.

## Контракт данных

Поле `confidence: Optional[str]` в `OCRRegistryEntry` уже существует (ADR-001). Формат значения теперь стандартизирован: `"{k}/{N}"`. Значения `null` и пустые строки рассматриваются как «не верифицировалось».

## Последствия

- `DocumentReader.verify()` перестаёт быть stub'ом.
- В `core/voting.py` выделены чистые функции нормализации и голосования — тестируются отдельно от OCR.
- `_resolve_entities` рефакторится: низкоуровневый OCR-проход выделен в `_ocr_pass(entries, chunk_size, max_workers) -> Dict[entity_id, Dict]`, `resolve` и `verify` используют его совместно.
- CLI: `verify --axes 1,3,5 --max-workers 5`.
- Env: `OCR_VERIFY_AXES` (default `"1,3,5"`).
- Доки: overview.md фиксирует стратегию; backlog.md закрывает пункт verify.

## Связанные решения
